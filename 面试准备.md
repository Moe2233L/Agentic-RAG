# AgenticRAG 面试准备

## 一、一句话说清项目

> 一个企业级 RAG 问答系统：上传文档 → 自动索引 → 用户提问 → 检索文档 → LLM 生成回答。支持深度 Agent 路由、知识图谱、联网搜索、SSE 流式。

---

## 二、技术栈（背熟）

| 技术 | 用途 | 一句话为什么选它 |
|------|------|------------------|
| FastAPI | 后端框架 | 异步支持好，写 SSE 方便 |
| ChromaDB | 向量库 | 轻量、免运维、嵌到项目里 |
| BGE-M3 | 嵌入模型 | 中文 SOTA、768 维省存储、本地免费 |
| BGE-reranker-v2-m3 | 重排 | cross-encoder 比向量检索准得多 |
| LangGraph | Agent 框架 | 有状态图，天然适合工具路由 |
| Neo4j | 图数据库 | 存实体关系，Cypher 查询 |
| Tavily | 联网搜索 | 给大模型用的搜索 API |
| Vue 3 + TypeScript | 前端 | 组合式 API 写起来快 |
| d3.js | 图谱可视化 | 力导向图开箱即用 |
| SSE | 流式通信 | 比 WebSocket 轻，单向推送够用 |

## 四、高频面试题

### 1. 为什么选 BGE-M3 不选 OpenAI embedding？
BGE-M3 在 C-MTEB 中文评测第一，768 维比 OpenAI 1536 维省一半存储和传输。本地部署免费，离线也能跑。换模型只改 `.env` 一行。

### 2. IP 还是 Cosine？选对了吗？
选错了。`"hnsw:space": "ip"` 对向量长度敏感，长文本得分虚高。应该用 Cosine 只比方向。改一行配置重建索引就行。

### 3. Reranker 干嘛的？为什么先搜 20 条再重排？
向量检索是 ANN 近似搜，召回率高但精度不够。cross-encoder 把 (query, doc) 逐对打分，精度高但算得慢。所以两步走：向量粗搜不遗漏 → reranker 精排取 Top 5。

### 4. LangGraph 怎么用的？不用行不行？
普通模式不用。deep mode 才走 LangGraph：route 节点 LLM 判断用哪个工具 → run 节点执行 → next 条件边循环。不用的话就 if-else 硬编码路由，加新工具得改代码。

### 5. SSE 流式怎么实现的？断了会重连吗？
FastAPI 用 `EventSourceResponse` 返回 SSE 流，前端用 `fetch.getReader()` 逐 token 读取。没实现重连——断了就断了。

### 6. 知识图谱真用了吗？效果如何？
建了图，插了数据，但 Cypher 查询是通用模板 `MATCH (n)... LIMIT 50`，和问题弱关联。应该用 LLM 生成 Cypher 但没实现。面试时主动承认这个不足。

### 7. 最大瓶颈在哪？
(1) ChromaDB 单机不抗分布式 (2) 无鉴权、无用户隔离 (3) session 内存泄漏 (4) 无测试 (5) BGE + reranker 单节点 GPU QPS 没测过

### 8. 怎么防止 prompt 注入？
没防。用户输入直接传给 LLM。这是待改进项。

### 9. LLM 抽取实体出错了怎么办？
没做校验。抽错就抽错，图谱里存的就是错的。唯一兜底是 reranker 可能把不相关的结果打低分。被动防御，不是主动纠错。

### 10. 对话历史怎么处理？
前端传 `conversation_id`，后端拼接最近 4 轮历史塞进 prompt。存在内存 dict 里，无 TTL 会泄漏。

## 五、主动说的亮点

1. **多路融合**：向量 + 图谱 + 联网三路检索，一路失败不影响其他
2. **两步精排**：ANN 粗搜 + cross-encoder 精排，比单阶段准
3. **Docker 一键部署**：docker-compose 起 backend + neo4j
4. **XSS 防御**：DOMPurify 净化 Markdown 输出

---

## 六、主动承认的不足

1. IP → Cosine 应该改
2. 知识图谱 Cypher 是模板，不是 LLM 生成
3. 无测试、无鉴权、无限流
4. 无 prompt 注入防御
5. 无 SSE 重连

> 面试官更欣赏「知道自己哪里不行」的人，而不是「硬说项目完美」的人。
